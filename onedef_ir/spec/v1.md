# onedef IR v1

This document is the canonical v1 contract for onedef IR JSON.

IR is a transport format, not a runtime API. It describes the HTTP contract that
SDK generators need:

- endpoint method, path, and success status
- path, query, header, and body request shape
- success response envelope and payload type
- non-success error body type
- group tree and group-level required header providers
- shared named model type table
- naming hints for target-language casing

## Top-Level Document

```json
{
  "version": "v1",
  "naming": {
    "initialisms": ["ID", "URL", "OAuth"]
  },
  "groups": [],
  "endpoints": [],
  "types": []
}
```

Rules:

- `version` is required and must be `"v1"`.
- `types` is required. Producers should emit an empty array when there are no
  named types.
- `naming`, `groups`, and `endpoints` are optional.
- Consumers must treat omitted or `null` arrays as empty arrays.
- Consumers must ignore unknown fields for additive compatibility.
- If both `groups` and top-level `endpoints` exist, consumers should process
  both. SDK structure should prefer `groups`.

## Naming

```json
{
  "initialisms": ["ID", "URL", "OAuth"]
}
```

`naming.initialisms` is a casing hint. SDK generators use it when converting IR
names into target-language identifiers.

Producers should trim empty entries, deduplicate case-insensitively, and sort
longer entries first when possible.

## Group

```json
{
  "id": "branch.booking",
  "name": "booking",
  "pathSegments": ["branch", "booking"],
  "requiredHeaders": ["X-Booking-Scope"],
  "providerHeaders": [
    {
      "name": "BookingScope",
      "wireName": "X-Booking-Scope",
      "type": { "kind": "string" },
      "required": true,
      "description": "Booking scope selector.",
      "examples": ["branch"]
    }
  ],
  "endpoints": [],
  "groups": []
}
```

Fields:

- `id`: stable group id. If omitted, consumers may use `pathSegments.join(".")`.
- `name`: required leaf name used as the source for SDK property/class names.
- `pathSegments`: exposed group path from the root. If omitted or empty,
  consumers may use `[name]`.
- `requiredHeaders`: compatibility string summary of headers introduced at this
  group boundary.
- `providerHeaders`: typed provider header contracts introduced at this group
  boundary. SDK generators should prefer this over `requiredHeaders`.
- `endpoints`: endpoints directly under this group.
- `groups`: child groups.

Group provider headers are not endpoint request fields. They describe ambient
values that a generated client should receive through a provider, constructor
argument, or header context. Child groups inherit ancestor provider headers.

## Endpoint

```json
{
  "name": "CreateBooking",
  "sdkName": "create",
  "method": "POST",
  "path": "/api/v1/branch/booking",
  "successStatus": 201,
  "requiredHeaders": ["Authorization", "X-Branch-Id", "Idempotency-Key"],
  "request": {
    "headerParams": [
      {
        "name": "IdempotencyKey",
        "wireName": "Idempotency-Key",
        "type": { "kind": "int" },
        "required": true
      }
    ],
    "body": { "kind": "named", "name": "CreateBookingRequest" }
  },
  "response": {
    "envelope": true,
    "body": { "kind": "named", "name": "Booking" }
  },
  "error": {
    "body": { "kind": "named", "name": "DefaultError" }
  }
}
```

Fields:

- `name`: required source endpoint name.
- `sdkName`: optional target SDK method name override.
- `method`: required HTTP method.
- `path`: required full HTTP path. Path variables use `{name}`.
- `successStatus`: required 2xx success status.
- `group`: optional flat grouping hint for documents without a group tree.
- `requiredHeaders`: optional complete header policy summary for this call.
  It contains inherited group provider headers plus endpoint-level required
  headers.
- `request`: required request contract.
- `response`: required success response contract.
- `error`: optional error contract. If omitted, consumers must use
  `DefaultError`.

`request.pathParams[].wireName` must match the variable names inside `path`.

## Request

```json
{
  "pathParams": [],
  "queryParams": [],
  "headerParams": [],
  "body": { "kind": "named", "name": "CreateBookingRequest" }
}
```

Fields:

- `pathParams`: required parameters used to fill `{variables}` in the path.
- `queryParams`: URL query parameters.
- `headerParams`: endpoint-level header parameters that the SDK method caller
  provides. Group provider headers must not appear here. If a producer has a
  typed source binding for the endpoint header, it should use that binding's
  `name`, `type`, and `required`; otherwise it should synthesize a required
  `string` parameter.
- `body`: request JSON body type. Omit it when there is no body.

## Parameter

```json
{
  "name": "ID",
  "wireName": "id",
  "type": { "kind": "uuid" },
  "required": true
}
```

Fields:

- `name`: source-language field or property name.
- `wireName`: path variable, query key, or header name.
- `type`: parameter type reference.
- `required`: whether the caller must provide the value.

Path parameters are always required.

## Response

```json
{
  "envelope": true,
  "body": { "kind": "named", "name": "Booking" }
}
```

Rules:

- `envelope: true` means the HTTP response body is a success envelope and the
  actual payload is inside `data`.
- `envelope: true` requires `body`.
- `envelope: false` means there is no success body.
- `204` success responses must use `envelope: false` and omit `body`.

Success envelope shape:

```json
{
  "code": "ok",
  "title": "OK",
  "message": "success",
  "data": {}
}
```

## Error

```json
{
  "body": { "kind": "named", "name": "DefaultError" }
}
```

Non-2xx responses do not use the success envelope.

`DefaultError` is a built-in named type. It may be referenced even when omitted
from `types`.

Default error shape:

```json
{
  "code": "invalid_body",
  "message": "request body is invalid JSON",
  "details": {}
}
```

Custom error policies should reference a custom named type and include that type
in `types`.

## TypeDef

```json
{
  "name": "Booking",
  "kind": "object",
  "fields": [
    {
      "name": "ID",
      "wireName": "id",
      "type": { "kind": "uuid" },
      "required": true
    }
  ]
}
```

Fields:

- `name`: unique type-table name.
- `kind`: currently `"object"`.
- `fields`: object fields.

`TypeRef.kind == "named"` usually references a `types[].name`. The built-in
exception is `DefaultError`.

## FieldDef

```json
{
  "name": "Profile",
  "wireName": "profile",
  "type": { "kind": "named", "name": "UserProfile", "nullable": true },
  "required": false,
  "nullable": true,
  "omitEmpty": true
}
```

Fields:

- `name`: source-language field/property name.
- `wireName`: JSON property name.
- `type`: field type reference.
- `required`: whether the JSON payload must contain this value.
- `nullable`: whether JSON `null` is allowed.
- `omitEmpty`: whether encoders may omit empty/null values.

## TypeRef

```json
{
  "kind": "list",
  "elem": { "kind": "named", "name": "Booking" }
}
```

Supported kinds:

| kind | JSON wire type | Notes |
| --- | --- | --- |
| `any` | any | dynamic JSON value |
| `bool` | boolean | |
| `int` | number | integer |
| `float` | number | floating point |
| `string` | string | |
| `uuid` | string | UUID string |
| `list` | array | requires `elem` |
| `map` | object | string keys only; requires `value` |
| `named` | object or alias | requires `name` |

`object` is used by `TypeDef.kind`, not by `TypeRef.kind` in v1.

## Producer Checklist

- Emit `version: "v1"`.
- Emit full endpoint paths.
- Emit one `request.pathParams` entry for every `{variable}` in the path.
- Include every named body/field/request/response/error type in `types`, except
  built-in `DefaultError`.
- Keep `types[].name` unique.
- Do not assign conflicting shapes to the same type name.
- Restrict map keys to strings.
- For group trees, put only newly introduced provider headers in each group's
  `providerHeaders`; keep the same wire names in `requiredHeaders` as a
  compatibility summary.
- For endpoint `requiredHeaders`, include inherited group headers and
  endpoint-level required headers.
- For `request.headerParams`, include only endpoint-level required headers.
  Use typed source bindings when present; synthesize required string params
  when absent.
- Represent 204 success as `response.envelope: false` with no body.

## Consumer Checklist

- Validate `version`.
- Normalize omitted or `null` arrays to empty arrays.
- Register built-in `DefaultError`.
- Build a `name -> TypeDef` table and reject duplicate type names.
- Validate every `TypeRef`.
- Traverse `groups` and top-level `endpoints`.
- Validate endpoint paths and path parameter matching.
- Decode non-204 success responses through the success envelope.
- Decode non-2xx error bodies using `error.body`, defaulting to `DefaultError`.

## Compatibility

v1 producers must not change the meaning of existing fields. Additive fields are
allowed. v1 consumers should ignore unknown fields and fail explicitly on unknown
type kinds.
