# onedef IR v1

This document is the canonical v1 contract for onedef IR JSON.

IR is a transport format, not a runtime API. It describes the HTTP contract SDK
generators need:

- endpoint method, path, and success status
- path, query, header, and body request shape
- success response envelope and payload type
- non-success error body type
- route tree and group-level header contracts
- shared named model table
- initialism hints for target-language casing

## Top-Level Document

```json
{
  "version": "v1",
  "initialisms": ["ID", "URL", "OAuth"],
  "routes": {
    "headers": [],
    "endpoints": [],
    "groups": []
  },
  "models": []
}
```

Rules:

- `version` is required and must be `"v1"`.
- `routes` is required and is the anonymous SDK route root.
- `models` is required. Producers should emit an empty array when there are no
  named models.
- `initialisms` is optional.
- Consumers must treat omitted or `null` arrays as empty arrays.
- Consumers must reject unknown fields in v1 documents.

## Routes

```json
{
  "headers": [],
  "endpoints": [],
  "groups": []
}
```

Fields:

- `headers`: global provider header contracts inherited by every endpoint and
  child group.
- `endpoints`: endpoints that generate methods directly on the SDK root client.
- `groups`: named child groups.

`routes` is the anonymous root of the SDK route tree. Route groups organize SDK
clients and inherited headers; they do not imply HTTP path prefixes. Each
endpoint `path` is always the full HTTP path.

## Initialisms

```json
["ID", "URL", "OAuth"]
```

`initialisms` is a casing hint. SDK generators use it when converting IR names
into target-language identifiers.

## Group

```json
{
  "name": "booking",
  "headers": [
    {
      "key": "X-Booking-Scope",
      "type": "string",
      "description": "Booking scope selector.",
      "examples": ["branch"]
    }
  ],
  "endpoints": [],
  "groups": []
}
```

Fields:

- `name`: required leaf name used as the source for SDK property/class names.
- `headers`: header contracts introduced at this group boundary. These are
  group-scope values that generated clients should receive through group
  configuration and inherit into child groups and endpoints.
- `endpoints`: endpoints directly under this group.
- `groups`: child groups.

Group `headers` entries are not endpoint request fields and do not have `name` or
`required`; their presence means they are required by this group scope.

## Endpoint

```json
{
  "name": "CreateBooking",
  "sdkName": "create",
  "method": "POST",
  "path": "/api/v1/branch/booking",
  "successStatus": 201,
  "request": {
    "headers": [
      {
        "name": "IdempotencyKey",
        "key": "Idempotency-Key",
        "type": "int",
        "required": true
      }
    ],
    "body": "CreateBookingRequest"
  },
  "response": {
    "envelope": true,
    "body": "Booking"
  },
  "error": {
    "body": "DefaultError"
  }
}
```

Fields:

- `name`: required source endpoint name.
- `sdkName`: optional target SDK method name override.
- `method`: required HTTP method.
- `path`: required full HTTP path. Path variables use `{name}`.
- `successStatus`: required 2xx success status.
- `request`: required request contract.
- `response`: required success response contract.
- `error`: optional error contract. If omitted, consumers must use
  `DefaultError`.

`request.paths[].key` must match the variable names inside `path`.

## Request

```json
{
  "paths": [],
  "queries": [],
  "headers": [],
  "body": "CreateBookingRequest"
}
```

Fields:

- `paths`: required parameters used to fill `{variables}` in the path. Path
  entries are always required.
- `queries`: URL query parameters. Query entries are optional by default.
- `headers`: endpoint-level header parameters that the SDK method caller
  provides for each call. Group `headers` entries must not appear here.
- `body`: request JSON body type. Omit it when there is no body.

## Parameter

```json
{
  "name": "ID",
  "key": "id",
  "type": "uuid"
}
```

Fields:

- `name`: source-language field or property name.
- `key`: path variable, query key, or header name.
- `type`: parameter type reference.
- `required`: request header parameters only; whether the caller must provide
  the header value.

## Response

```json
{
  "envelope": true,
  "body": "Booking"
}
```

Rules:

- `envelope: true` means the HTTP response body is a success envelope and the
  actual payload is inside `data`.
- `envelope: true` requires `body`.
- `envelope: false` means there is no success body.
- `204` success responses must use `envelope: false` and omit `body`.

## Error

```json
{
  "body": "DefaultError"
}
```

Non-2xx responses do not use the success envelope.

`DefaultError` is a built-in named model. It may be referenced even when omitted
from `models`.

## ModelDef

```json
{
  "name": "Booking",
  "kind": "object",
  "fields": [
    {
      "name": "ID",
      "key": "id",
      "type": "uuid",
      "required": true
    }
  ]
}
```

Fields:

- `name`: unique model-table name.
- `kind`: currently `"object"`.
- `fields`: object fields.

## FieldDef

```json
{
  "name": "Profile",
  "key": "profile",
  "type": "UserProfile?",
  "required": false,
  "omitEmpty": true
}
```

Fields:

- `name`: source-language field/property name.
- `key`: JSON property name.
- `type`: field type reference.
- `required`: whether the JSON property is required.
- `omitEmpty`: whether null values should be omitted by SDK encoders.

Value nullability is expressed only by the type expression suffix `?`.

## Type Expression

```json
"map<string, list<Booking>>"
```

Type references are string expressions.

| expression | meaning |
| --- | --- |
| `any` | arbitrary JSON value |
| `bool` | boolean |
| `float` | floating point number |
| `int` | integer |
| `string` | string |
| `uuid` | UUID string |
| `Booking` | named model reference |
| `list<Booking>` | array |
| `map<string, Booking>` | string-keyed object |
| `Booking?` | nullable named value |
| `list<Booking?>` | non-null list with nullable items |
| `list<Booking>?` | nullable list |

`map` keys must be `string`. Nested expressions are allowed, for example
`map<string, list<Booking>>`.

## Producer Rules

- Include every named body/field/request/response/error type in `models`, except
  the built-in `DefaultError`.
- Keep `models[].name` unique.
- Emit one `request.paths` entry for every `{variable}` in the endpoint path.
- Root-scope headers belong in `routes.headers`.
- Group-scope headers belong in `routes.groups[].headers` and nested
  `groups[].headers`.
- Endpoint call-specific headers belong in `request.headers`.
- Do not emit derived header summaries; consumers can derive effective headers
  from ancestor group `headers` plus endpoint `request.headers`.
