# onedef v0.2.0 Test Plan

## Context

`fiber`의 테스트 구성은 단순 유닛 테스트보다 public contract를 실제 요청과 조합으로 반복 검증하는 쪽에 가깝다. 주요 특징은 다음과 같다.

- Core app, router, binder, client, middleware, storage, log 등 패키지 경계별 테스트 파일을 둔다.
- `httptest`, `app.Test`, 실제 listener, table-driven subtest로 HTTP contract를 검증한다.
- 정상 경로뿐 아니라 invalid input, panic, timeout, duplicate config, security regression, concurrency, benchmark까지 테스트 축을 반복한다.
- CI에서 `-race`, `-shuffle=on`, 반복 실행, OS/Go version matrix를 사용해 flaky/racy test를 빨리 드러낸다.

`onedef`는 현재 `internal/app`, `internal/inspect`, `ir`, public external API 쪽에 좋은 출발점이 있지만, framework 수준의 신뢰를 만들기에는 runtime matrix, IR/SDK golden, regression/security, concurrency, fuzz/benchmark 축이 아직 얇다.

## P0: Framework Core Contract

깨지면 바로 프레임워크 신뢰가 무너지는 영역이다. v0.2.0 테스트 보강은 여기서 시작한다.

### Runtime HTTP Contract

- Success envelope: `200`, `201`, body shape, JSON encoding.
- No content: `204` status는 empty body를 유지한다.
- Error envelope: default error body, status code, code/message/details.
- Invalid body: malformed JSON, empty body, unknown body shape.
- Path parameter conversion: string, int variants, uint variants, bool, uuid, pointer fields.
- Query parameter conversion: missing optional values, invalid scalar, repeated values policy.
- Header parameter conversion: required, optional pointer, missing required, invalid scalar.
- Method and route behavior: registered method, unregistered method, not found route.

Candidate files:

- `internal/app/app_test.go`
- `internal/app/make_handler_func_test.go`
- `internal/app/convert_path_value_test.go`

### Endpoint Reflection Validation

- Missing or invalid method marker.
- Invalid `path` tag: empty where not allowed, missing slash, malformed variables.
- Invalid `status` tag: non-integer, non-2xx, `204` with non-empty response.
- Missing `Request` or `Response` field.
- Handler signature validation: must implement `Handle(context.Context) error`.
- Request field classification: path, query, body, header.
- Unsupported field types and unsupported map keys.
- Header tag validation: unexported field, empty tag, duplicate header by case-insensitive name.

Candidate files:

- `internal/inspect/inspect_test.go`
- `ir/parse_test.go`

### Group, Dependency, Middleware

- Nested group path joining.
- Endpoint leaf path behavior under groups.
- Required header inheritance and endpoint header conflict.
- `OmitHeader("Authorization")` behavior and invalid omit cases.
- Dependency inheritance, child shadowing, endpoint shadowing.
- Missing dependency fails before serving.
- Duplicate dependency binding in the same scope fails.
- Middleware order: parent, child, endpoint.
- Middleware skip and re-add by name.
- Nil middleware and nil returned handler fail fast.
- Recover middleware converts panic into internal error.
- Error policy inheritance and endpoint override.

Candidate files:

- `internal/app/app_test.go`
- `internal/app/middleware_test.go`
- `internal/meta/dependency_test.go`
- `internal/meta/middleware_test.go`
- `internal/meta/error_policy_test.go`

### CI Baseline

- Add repository test workflow.
- Run `go test ./... -race -shuffle=on -count=1`.
- Use local module/cache paths in CI only if needed.
- Keep generated/example cache directories out of the test package list.

## P1: IR and SDK Contract

사용자가 직접 소비하는 generated contract를 고정하는 영역이다.

### IR Golden Tests

- Simple endpoint spec.
- Grouped endpoint spec.
- Nested group tree.
- Required provider headers and endpoint-specific headers.
- Optional/nullability/type mapping.
- Initialisms metadata.
- Named vs anonymous request/response types.
- Unsupported type error messages.
- Deterministic ordering for endpoints, types, headers, groups.

Candidate files:

- `ir/parse_test.go`
- `internal/app/grouped_sdk_spec_test.go`
- `testdata/ir/*.golden.json`

### Dart SDK Golden Tests

- Generated package file tree.
- Root client and nested group clients.
- Provider constructor shape.
- Endpoint method names, including `SDKName`.
- Header arguments and provider inheritance.
- Request/response model names and JSON helpers.
- Error/default transport contract.
- Generated SDK compiles or passes Dart analyzer/test.

Candidate files:

- `generators/dart/packages/onedef_gen/test/generator_test.dart`
- `generators/dart/packages/onedef_core/test/onedef_core_test.dart`
- `generators/dart/packages/onedef_gen/test/fixtures/*.json`

### Public API Compatibility

- External package tests for `onedef.Group`, `Endpoint`, `Endpoints`, `Dependency`, `Middleware`, `ErrorPolicy`, `GenerateIRJSON`, `GenerateSDK`.
- Compile-only tests for common user-facing examples.
- Ensure runtime-only nodes such as dependencies and middleware do not leak into IR/SDK.

Candidate files:

- `public_external_test.go`
- `middleware_external_test.go`
- `definition_test.go`

## P2: Regression, Security, Edge Cases

Fiber식 battle-tested 감각을 만드는 영역이다. 한 번 발견한 버그는 전부 테스트로 봉인한다.

### HTTP and Header Edge Cases

- Header name normalization and duplicate detection.
- Unsafe header values.
- Case-insensitive auth omit behavior.
- Empty, whitespace, and non-ASCII header values where applicable.
- Path parameter with encoded values.
- Query values with empty string, zero, false, and repeated keys.

### Security Regression Tests

- Internal errors must be masked by default.
- Custom `HTTPError` data is passed through only when intended.
- Panic recovery never exposes panic value.
- Generated SDK must not expose server dependency details.
- Malformed endpoint definitions fail at startup, not first request.

### Concurrency and Isolation

- Concurrent requests to the same endpoint do not share endpoint instance state.
- Dependency values are shared intentionally, request structs are not.
- Middleware call recording is isolated per test/request.
- Group registration and serving behavior is deterministic.
- Race detector should stay clean under representative concurrent HTTP tests.

### Example E2E

- `example/hello_world` runtime smoke.
- `example/e2e` backend plus generated Dart client smoke.
- `example/salon_operations` interface/spec generation smoke.

## P3: Performance, Fuzzing, Long-Term CI

장기적으로 프레임워크 유지 비용을 낮추는 영역이다.

### Benchmarks

- `convertPathValue` for scalar and uuid types.
- `MakeHandlerFunc` request handling for success and validation errors.
- `ir.Parse` on small, medium, and grouped specs.
- Grouped SDK spec generation.
- Dart generator render package on representative specs.

### Fuzz Tests

- Path tag and path parameter parsing.
- Header tag parsing.
- Scalar conversion from raw path/query/header strings.
- IR type extraction for nested structs, pointers, slices, maps.
- Initialism normalization.

### CI Hardening

- Add repeated test job with `-race -shuffle=on -count=10` or higher.
- Add OS matrix after P0 stabilizes.
- Add Go version matrix for supported versions.
- Add coverage artifact.
- Add benchmark workflow for non-blocking regression visibility.
- Add Dart test/analyze workflow for generator and core packages.

## Suggested Rollout

1. P0 runtime/reflection/group tests and baseline CI.
2. P1 IR/SDK golden tests and public API compatibility expansion.
3. P2 regression/security/concurrency tests from real bugs and risky edge cases.
4. P3 fuzz, benchmark, repeated CI, and OS/version matrix.

The guiding rule is simple: every public onedef contract should have at least one happy path, one invalid input path, and one composition test with group, dependency, middleware, or SDK generation when applicable.
